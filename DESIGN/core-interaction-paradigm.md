# 核心交互范式: 导演模式

## 概述

用户不扮演任何角色, 不出现在剧情中。用户的指令相当于推动故事朝某个方向发展, AI 负责生成叙事内容。

本质是一个 **Director → Narrator** 的单向调度关系: 用户是"导演", 通过指令调度故事; AI 是"叙述者", 生成叙事响应。

## 界面布局

对话式界面, 导演指令与叙事输出在同一对话流中线性呈现。侧边栏放置状态栏、生效指令、人物列表等辅助信息。

```
┌──────────────────────────────┬───────────────────┐
│                              │  状态栏            │
│   对话区 (叙事流)              │  ┌──────────────┐ │
│                              │  │ 理智值: 45    │ │
│   AI: 雨夜的巷子里...          │  │ 阶段: 动摇     │ │
│                              │  │ 好感度: 30    │ │
│   导演: 让主角发现一封密信      │  │ 阶段: 敌对     │ │
│                              │  └──────────────┘ │
│   AI: 主角在地下室角落...      │                   │
│                              │  生效指令          │
│                              │  ┌──────────────┐ │
│                              │  │ 🎭 悬疑氛围    │ │
│                              │  │   (剩余 3 轮)  │ │
│                              │  │ 🚫 不能出现    │ │
│                              │  │   新人物(2轮)  │ │
│                              │  └──────────────┘ │
│                              │                   │
├──────────────────────────────┤  人物列表          │
│ 导演输入框                     │  ...              │
└──────────────────────────────┴───────────────────┘
```

### 设计理由

对话区保持线性叙事流, 保证用户阅读的连贯性。导演指令和叙事输出在同一流中交替出现, 但在数据模型中是不同类型的回合 (导演回合 / 叙事回合), 便于精确回滚、重试和审计。

## 指令类型

### 分类: 项目设定 vs 导演指令

两类东西需要明确区分:

**项目设定** — 创建项目时定义的固有属性, 极少变更, 变更意味着修改项目本身:

| 类型 | 示例 | 说明 |
|------|------|------|
| 世界观概述 | "这是一个蒸汽朋克世界..." | 创建项目时编写 |
| 叙事风格 | "第二人称, 现在时, 悬疑笔法" | 创建项目时编写 |
| 永久约束 | "主角不能死亡" "不能打破第四面墙" | 创建项目时设定, 运行时也可追加 |

**导演指令** — 对话过程中随时发出, 驱动故事推进:

| 类型 | 示例 | 作用 | 生命周期 |
|------|------|------|---------|
| 剧情指令 | "让主角发现一封密信" | 直接注入叙事事件 | 一次性消费 |
| 基调指令 | "保持悬疑氛围" | 约束叙事风格 | 带 TTL, N 轮后自动移除 |
| 临时约束 | "接下来不能出现新人物" | 短期硬性规则 | 带 TTL 或手动撤销 |
| 元指令 | "跳过日常, 直接到冲突" | 时间轴 / 节奏控制 | 不进 Narrator 上下文 |
| 修正指令 | "刚才不合理, 重来" | 回滚 + 重新生成 | 不进 Narrator 上下文 |

### 永久约束的运行时追加

用户在运行时可将临时约束"提升"为永久约束 (如侧边栏点击"设为永久")。提升后该约束从 Active Directives 移入 System Prompt, 对整个故事生效。

### 元指令与修正指令的处理

元指令和修正指令不直接进入 Narrator 的上下文, 由 Orchestrator 拦截并触发特殊流程:

- **元指令**: Orchestrator 让 Memory Agent 压缩当前场景 → 推进时间线 → 以新场景开启 Narrator 的新一轮生成。Narrator 看到的是一个普通的导演输入, 但剧情已经跳跃。
- **修正指令**: Orchestrator 从历史中移除最近一轮 → 将修正意见作为附加约束注入 → 重新生成。对 Narrator 而言, 上一轮"不存在"。

## 回合模型

每次导演指令触发一个"导演回合", AI 的叙事响应是一个"叙事回合"。两者在数据模型中是不同类型:

```
[导演回合] "让主角发现一封密信"
    │
    ▼ (经过 Agent 编排流水线)
[叙事回合] "主角在地下室角落翻到了一封泛黄的信..."
    │
    ▼ (用户阅读后发出下一条指令)
[导演回合] "密信内容提到三年前的港口大火"
    ...
```

回合模型的优点:
- 回滚和重试精确到"叙事回合", 而非单条消息
- 审计系统只审计叙事回合内容, 不审计导演指令
- 压缩以回合为单位, 便于管理
